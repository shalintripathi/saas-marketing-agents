# Handoff Protocols: Agent-to-Agent Work Transfer

## Overview

The handoff protocol ensures clear, documented transfer of work from one agent (or phase team) to the next. Handoffs are the critical points where miscommunication happens, so this document standardizes the process across all CATALYST phases.

A well-executed handoff includes:
1. **Clear specifications** (what exactly are you handing off?)
2. **Acceptance criteria** (how does the receiving agent know it's right?)
3. **Context documentation** (why this matters, what decisions led here)
4. **Feedback loop** (how does the receiving agent inform the upstream agent of results?)

## Phase-Specific Handoff Checklists

### Phase 0 → 1 Handoff Checklist

**Discovery Phase Lead ensures:**
- [ ] Market research completed (at least 10 customer interviews)
- [ ] Competitive landscape mapped (3-5 competitors analyzed)
- [ ] Personas drafted (firmographic + psychographic data)
- [ ] SEO audit completed (technical + keyword opportunity)
- [ ] Analytics baseline established (historical performance documented)
- [ ] Key assumptions documented for validation
- [ ] Handoff spec completed and approved

**Phase 1 Lead confirms:**
- [ ] All Phase 0 deliverables received and reviewed
- [ ] No unclear findings (asks for clarification)
- [ ] Data points marked for validation during optimization (Phase 5)
- [ ] Ready to begin strategy workshop

**Quality Gate Criteria:**
- [ ] Customer research validated with 8+ interviews
- [ ] Personas include all 6 data elements
- [ ] Competitive differentiation identified
- [ ] TAM/SAM/SOM calculated
- [ ] Technical SEO audit completed
- [ ] Key assumptions listed

## Common Handoff Issues & Solutions

### Issue: Receiving Agent Doesn't Understand Deliverables

**Prevention:**
- Spend 30 minutes in kickoff explaining context
- Use simple language, avoid jargon
- Share examples of what "done" looks like

**Solution:**
- Schedule clarification call with sending agent
- Have phase lead mediate if persistent confusion
- Document clarifications for future reference

---

### Issue: Deliverables Don't Meet Acceptance Criteria

**Prevention:**
- Review acceptance criteria before handoff
- Get preliminary feedback 1 week before formal handoff
- Have quality review with peer

**Solution:**
- Sending agent revises (2-3 days)
- Receiving agent reviews revised version
- Iterate until criteria met

---

### Issue: Receiving Agent Can't Access Deliverables

**Prevention:**
- Share links in handoff spec, not just file names
- Verify access permissions before handoff
- Test links the day before handoff

**Solution:**
- Sending agent re-shares links with correct permissions
- Check folder structure (receiving agent may be looking in wrong place)
- Consider downloading and re-uploading if access issues persist

---

### Issue: Handoff Timeline Slips

**Prevention:**
- Build buffer into timelines (5-10% padding)
- Communicate about blockers early
- Have contingency plan (what ships if we're delayed?)

**Solution:**
- Extend downstream phase timeline (cascade delay)
- Ship MVP of downstream phase (do essential work, defer nice-to-haves)
- Add resources to catch up
- Escalate if delay significant

---

### Issue: Receiving Agent Disagrees with Sending Agent's Work

**Prevention:**
- Include feedback from receiving agent during production (not after)
- Get buy-in from key stakeholders during phase
- Frame work as "draft" with iteration expected

**Solution:**
- Have collaborative discussion (not adversarial)
- If fundamental disagreement: Escalate to phase leads
- Document decision and rationale
- Move forward (don't get stuck in analysis)

## Best Practices for Smooth Handoffs

### 1. Document Everything

Don't rely on verbal communication for critical info. Use the handoff spec.

### 2. Handoff Early

Don't wait until the very end to inform receiving agent. Brief them mid-phase.

### 3. Iterate Before Final Handoff

Get feedback from receiving agent while you can still revise (not after).

### 4. Over-Communicate

Send more updates than seems necessary. It prevents surprises.

### 5. Make It Easy

Provide links, not just file names. Provide context, not just deliverables. Provide examples, not just requirements.

### 6. Assume Receiving Agent is Busy

Make it easy for them to understand quickly. Your 30-minute explanation might be their only chance to learn before they have to act.

### 7. Follow Up

1 week after handoff, check in: "Are you unblocked? Do you have what you need?"

## Summary

Handoff protocols exist to:
1. **Reduce misunderstanding** (clear specs prevent surprises)
2. **Enable coordination** (everyone knows dependencies)
3. **Maintain quality** (acceptance criteria ensure standards)
4. **Enable feedback** (loops prevent repeating mistakes)
5. **Build institutional knowledge** (documentation helps future campaigns)

Use these protocols for every handoff in CATALYST. Over time, handoffs become faster as teams internalize the process and understand each other's needs.
